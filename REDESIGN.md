# ShadowVerse · 产品与前端重设计（2026-06）

> 本文是这一轮"重新设计"的**单一事实源**。它不替代 `ARCHITECTURE.md`（引擎地基仍以那份为准），而是回答另一个问题：
> **引擎已经够强了，为什么用起来还是不对？接下来该往哪走？**
>
> 配套产物：`docs/ui-prototype.html`（可点击的前端原型，假数据，本文的视觉与交互参照）。

---

## 一、缘起：v1 哪里不对

引擎不是问题。10 天里它长出了五层架构、37 模块、59 个 MCP 工具、测试全绿、完整网页控制台——事件溯源、状态重建/记忆检索分离、anchors 身份核心、成长时刻、跨世界魂、11 模式、世界书触发引擎、ST 导入、多模态……该有的都有，甚至**过剩**。

真实使用过（十几个世界/角色），体验却"就是不太好"，前端"太普通没感觉"。把痛点拆开是三条，且都指向同一个病根：

- **角色像通用 AI** —— anchors 太抽象，喂给模型的 prompt 太薄，压不住基座模型的通用腔。
- **故事推不动** —— 没有戏剧压力，开场是空白框，没有带前提的入口。
- **机制感觉不到** —— 记忆、成长、羁绊、跨世界全是后台数据结构，从没在体验里"落到心上"。

**病根：项目把几乎全部力气投在"确定性状态引擎"上，而决定魅力的那一半——生成的手艺、戏剧、声音、把机制做成"被感觉到"的表面——几乎是空的。** 这恰恰被架构的立国之本"Model A：引擎管状态，智力=宿主模型"注定了：它从设计上就把"决定魅力的那一半"外包了出去。

对照产品 VisionTale（同类酒馆）得到的结论更扎心：**它在架构上几乎和我们一样、甚至更少，但它赢在手艺与呈现**——被设计过的排版开场（LIFEPATH 式开局）、封面氛围、内置文风、把写作手艺做成可勾的"开关"（活人感加强 / 对白优化 / 文风·五感 / 防超雄……）。而这些，**我们的引擎本来就支持（greeting / preset / regex / worldbook / 变量），只是从没有人坐下来、带着品味把它做出来、摆出来。**

> 一句话：**v1 的失败是实质（魅力·手艺）问题，不是架构问题。**

## 二、北极星（终极目标）

产品层（对用户）：

> **让角色在世界中经历故事，并把经历留下，成为回响。**

它是一个统一的叙事底座，把三件事融在一起、并随自主性递增而演化：

```
创作  →  推演  →  记录
（你造世界/角色） （世界与角色自行发展） （一切沉淀为可读、可扩展的作品）
```

工程层（对内核，沿用 ARCHITECTURE）：

> **一切皆事件，世界/角色/关系/记忆/故事皆为其投影。**

两层不矛盾，但**铁律是：内核绝不许浮上来**。用户认识的是"她经历过什么""你们之间发生过什么"，不是"她的事件流"。

### 检验闸门（每加一个设计都过一遍）

- 产品层：它在提升 **(A) 角色真实感 / (B) 世界沉浸感 / (C) 长期成长感** 吗？都不是 → 大概率偏了。
- 内核层：它让那三种感觉**更便宜、更可靠**吗？

---

## 三、最重要的两条判断（决定取舍）

1. **差异化只有一个：跨世界回响（暗宇宙）。** 跑卡（酒馆 RP）是 commodity——一张好卡 + 一套好预设就有，且能从社区导入。酒馆和 VisionTale 给不了的，只有"**同一个角色跨世界地存在、记得、改变**"，以及由此而来的"**离别 / 回响**"。这是唯一值得重锤的territory。它在工程上只是 fork，**在产品上必须是一场离别**。

2. **手艺 > 架构。别再深化系统。** 该做的不是再加机制，而是：(a) 把已建好的能力**做成被感觉到的表面**；(b) 真正坐下来做**一个完成度高的作品**（世界 + 有声音的角色 + 排版开场 + 文风预设 + 封面）。

> ⚠️ 最大的复发风险：**表面跑到实质前面**。原型越精致，越要记得——mockup 证明不了"有没有一个值得回去见的角色"，而那正是 v1 沉掉的地方。

## 四、产品结构（用户视角的信息架构）

顶层导航只暴露用户心智里的东西，工程概念（世界/实体/魂/事件流）收进后台：

```
对话(酒馆RP)  陪伴(虚拟女友)  小说(创作)  作品  角色  [资源▾]  设置
                                                         └ 世界书 · 素材 · 预设
```

- **三种核心玩法 = 同一底座的三种关系密度**：酒馆（低密度·多角色碰撞）／陪伴（高密度·单角色深关系）／小说（创作·读与记录）。底层共享角色状态与关系；切换的是"控制权 + 关系密度"，不是三套系统。
- **作品（Works）**：带封面的世界卡，一进门就有氛围。一个作品 = 一个世界 + 一套角色，你用任意"玩法/镜头"进去体验。
- **角色（Characters）**：可跨作品存在的人；跨世界的角色挂 `✦` 角标。
- **资源**：世界书 / 素材库 / 预设——创作原料，收纳进一个下拉，避免台面变回"什么都堆的 dashboard"。
- **看板娘（星瞳）**：统一的拟人化入口，平时是右下角小标，展开是抠图立绘 + 悬浮选项 + 一行控制台（Token / 上下文% / 世界书在场数）。是向导 + 仪表盘，但不能喧宾夺主。

### 暗宇宙 / 跨世界（差异化，必须被感觉到）

- **回响 / 离别**：把角色带出一个世界 = 一场全屏仪式场景（她留一句话、新世界带着旧记忆开场）。这是产品里情感浓度最高的一刻，**该得到最多手艺**，不是一个小按钮。
- **星图**：世界节点 + 连接边 + 跨世界角色标记——一张图把"暗宇宙"立住。
- **编年史**：一个世界跨所有故事线的事件 + ★ 成长时刻 + 跨世界去向，合到一条时间轴。
- **化身对照**：同一个灵魂在不同世界的样子/记忆并排看（incarnations 数据已有）。

## 五、三种核心体验（前端要点）

**对话 / 酒馆 RP**
- 排版过的开场（引子 + LIFEPATH 式开局选择），不是空白框。
- 头像气泡；swipe 一楼多候选（◀ 2/3 ▶）；快速回复 chips；多角色"在场"= 群聊。
- 角色输出的 HTML 状态面板（HUD）在沙箱里渲染。
- 右侧面板四 tab：作品 / 开关 / 参数 / 角色（含**当前驱动 Active Drives**，露出 Desire 层）。

**陪伴 / 虚拟女友**
- 单角色的"家"：立绘 + 此刻状态 + 关系四轴（好感/心动/信任/心防）+ 关系阶段。
- **她记得**（从共同经历唤起的一句）、**她主动发来的消息**、**重要时刻时间线**、**身份锚点**（不会变的内核）。
- 底部即"✦ 回响——带她离开这个世界"的入口（→ 离别场景）。

**小说 / 创作**
- 排版考究的章节阅读器（衬线、缩进、行距）。
- 写→审→改→落产线状态；钩子台账（α 悬念 / 伏笔回收）；线分支 · 蝴蝶效应；质检/反思报告；一稿多吃（→ CYOA/剧本/漫画/跑团，走 convert）。

---

## 六、创作者层（按需出现，不破坏沉浸）

这些是给"创作者/调试"的，做成悬停/开关/面板，绝不糊在沉浸阅读界面上：

- **导演指令（OOC 通道）**：输入框上的开关，开启后你输入的是给系统的指令（`[下一幕下雨]`/`[她更冷淡些]`/`[跳到第二天]`），与"对她说话"明确分开。
- **记忆溯源（Context Source）**：每条回复可展开——本回合触发了哪些世界书条目、检索到哪段记忆（带权重）、哪条锚点在起作用。把看不见的机器摊开。
- **自动提取至世界书**：对话里冒出的新设定，一点 → AI 填好名称/关键词/描述。让"推演→记录"自动化。
- **控制台**：Token 消耗 / 上下文占用 / 世界书在场数。

## 七、前端设计语言

- **暗色电影感**，不是 dashboard：深底 + 极淡星云光晕，紫（`--violet`）为主、青/金/玫瑰为辅，封面与立绘带氛围。
- 圆角、柔边、留白、悬浮微动；避免统计 chip 堆砌和工程词上墙。
- **响应式适配**：窄屏（≤1024）自动收侧栏、改单列。
- 具体 token / 组件以 `docs/ui-prototype.html` 为参照。

## 九、红线 / 陷阱

- ❌ **别让表面超过实质**——再精装修，台上没戏也是空的。每往前端加东西，先问是否在为"一个值得回去见的角色"服务。
- ❌ **别把差异化拍平**——陪伴 + 跨世界回响是情感重心，跑团/音乐/教育/梦境是"也能玩"，不该同等权重。
- ❌ **内核别浮上来**——事件流/投影/谱系这些词不进用户界面。
- ❌ **别再深化引擎**——它已经过剩；加功能=重蹈 v1 覆辙。
- ⚠️ **离别要做成一场戏**，不是一个按钮。

## 十、路线 / 下一步

原型方向已收敛、可**定稿**，用来指导落地。接下来二选一：

1. **落地前端**：照 `docs/ui-prototype.html` 重写真实 `sv/web/index.html`，对接现有 `/api`，引擎一行不动。
2. **（更接近北极星）先让一个作品真的活一次**：把《夜行动物》/苏屿配一套好预设，在引擎里端到端跑几轮，标准只有一条——**你还想不想接着看下去。** 这是 mockup 永远答不了、却决定一切的问题。

> 建议：哪怕先落地一部分前端，也要尽快回到第 2 件事。**v1 沉在实质，新版别再沉一次。**
